# 0006 - Organization Vault Access And Key Rotation

## Status

Accepted.

## Context

Umbra supports personal vaults and team vaults. A team needs organization-level membership for administration, but crypto access must stay scoped to individual vaults.

If organization membership automatically opened all organization vaults, a future private project vault inside the same organization would require redesign. If vault membership existed without vault key wrapping, the server could authorize access that the client could not decrypt.

## Decision

Personal vaults use `org_id = null`.

Organization vaults use `org_id = org_id`, but organization membership does not automatically grant vault decryption.

Vault access requires:

- active `vault_members` membership;
- server role permission for the action;
- active `vault_key_wrappings` envelope for the current vault key generation.

Vault key wrappings are encrypted by the granting client. The server stores the envelope as JSON and never sees the vault key.

Each wrapping using `user_public_key` contains a fresh `ephemeral_public_key`. The matching ephemeral private key is generated by the granting client, used once to derive the wrapping key, and discarded.

Removing a user from a vault:

- marks the member as removed;
- revokes active key wrappings for that user;
- sets `vaults.needs_key_rotation = true`.

Rotating a vault key:

- increments `current_key_generation`;
- revokes old active wrappings;
- stores new wrappings for remaining members;
- stores re-encrypted item revisions for the new generation;
- clears `needs_key_rotation`.

## Consequences

The first server implementation can enforce sync visibility and role checks without decrypting secrets.

Removed members stop receiving future sync immediately, but secrets they already saw remain compromised until the real underlying credentials are rotated. Vault key rotation prevents future item revisions from being decrypted with the old vault key.

The client must implement wrapping creation, item re-encryption, and rotation submission. The server only validates membership, generation transitions, and revision conflicts.
